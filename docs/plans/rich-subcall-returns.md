# Rich Subcall Returns (Episodic Memory)

## Status: Planned

## Context

When `llm_query` spawns a child client at depth>1, the parent receives only the final answer string. It is blind to what the child explored, tried, and what failed — Slate calls this "blind N-step execution." Returning richer metadata (an "episode summary") alongside the answer enables the parent model to course-correct based on what the child discovered.

## Problem

```ruby
# Current: parent gets only a string
response = child.completion(prompt: sub_prompt).response
```

The parent has no visibility into:
- What code the child executed
- What errors it encountered
- How many iterations it took
- Whether it was forced to a final answer
- Key intermediate findings

## Architecture

### Option A: Structured Return from `llm_query`

Change `llm_query` to return a richer object instead of a plain string when `depth < max_depth` (recursive path). The REPL helper would return a hash or struct:

```ruby
{
  answer: "the final answer string",
  episode: "Compressed summary: explored X, found Y, failed at Z...",
  iterations: 3,
  forced_final: false
}
```

The episode summary would be generated by the child client's compaction infrastructure (reusing `CompactionPrompt`) applied to the child's full conversation history before returning.

### Option B: Episode as Metadata Only

Keep `llm_query` returning a string for backward compatibility, but attach the episode summary to the `CompletionResult` metadata. The parent client extracts it from the child's result and includes it in the feedback message.

**Recommended: Option A** — the model needs to see the episode to make informed decisions. Returning it as part of the `llm_query` return value makes it directly accessible in REPL code.

## Implementation Steps

### 1. Add `episode_summary` generation to `Client#completion`
After the completion loop ends (before building `CompletionResult`), use the compaction infrastructure to compress the full message history into a summary string. Store it as `metadata[:episode_summary]`.

### 2. Modify `Client#llm_query` recursive path
Instead of `child.completion(prompt: sub_prompt).response`, capture the full `CompletionResult` and return a structured hash when the child has episode data.

### 3. Update REPL helper documentation
The system prompt's documentation for `llm_query` should note that it can return a hash with `answer` and `episode` keys when recursion depth allows.

### 4. Backward compatibility
When `max_depth <= 1` (single-shot fallback), `llm_query` continues returning a plain string. The model already handles this path. Add a guard in the REPL helpers to handle both return types gracefully.

### 5. Tests
- Unit test: child client generates episode summary
- Integration test: parent receives episode data from `llm_query`
- Backward compatibility: single-shot fallback still returns string
